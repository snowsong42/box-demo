# LovyanGFX Sprite 图像加载指南 (ESP-IDF)

> 总结自 box-demo 项目开发过程中遇到的问题与解决方案

## 1. 核心问题：SPIFFS VFS 与 LovyanGFX DataWrapper 不兼容

### 现象

`drawPngFile("/spiffs/xxx.png", x, y)` 和 `drawJpgFile("/spiffs/xxx.jpg", x, y)` 在 ESP-IDF 下返回 `false`，图片不显示。

### 根因

LovyanGFX 的 `drawPngFile(const char* path)` 内部使用 `_create_data_wrapper()` 创建文件读取器，该读取器在 Arduino 环境下通过 `SPIFFS` 对象工作，但在 ESP-IDF 原生环境下无法正确处理 SPIFFS VFS 路径。

### 解决方案

用 C 标准库 `fopen`/`fread` 读取文件到堆缓冲区，再调用基于内存的 `drawPng(buf, len, x, y)` / `drawJpg(buf, len, x, y)`：

```cpp
FILE* fp = fopen("/sdcard/image.png", "rb");  // 资源已迁至 SD 卡
fseek(fp, 0, SEEK_END);
long fsize = ftell(fp);
fseek(fp, 0, SEEK_SET);
uint8_t* buf = (uint8_t*)malloc(fsize);
fread(buf, 1, fsize, fp);
tft.drawPng(buf, fsize, x, y);  // 或 spr.drawPng(buf, fsize, 0, 0)
free(buf);
fclose(fp);
```

## 2. Sprite 内存与 PSRAM

### LovyanGFX Sprite 内存分配策略

`createSprite(w, h)` 内部（`Panel_Sprite.cpp`）：

```cpp
// 优先 PSRAM（若 psram 标志为 true）
_img = heap_caps_malloc(len, MALLOC_CAP_SPIRAM | MALLOC_CAP_8BIT);
// 回退到内部 DMA RAM
if (!_img) _img = heap_caps_malloc(len, MALLOC_CAP_8BIT | MALLOC_CAP_DMA);
```

### 内存计算

| 精灵尺寸  |    色深    |     每帧大小     |
| --------- | :---------: | :---------------: |
| 200×200  | RGB565 (2B) |  **80 KB**  |
| 500×150  |   RGB565   | **150 KB** |
| 28 帧 GIF |      -      | **2.24 MB** |

**ESP32-S3 内部 DRAM：~320 KB**（自由堆），28 帧无法全部容纳。

### 必须启用 PSRAM

在 `sdkconfig` 中：

```
CONFIG_SPIRAM=y
CONFIG_SPIRAM_MODE_OCT=y
CONFIG_SPIRAM_SPEED_80M=y
CONFIG_SPIRAM_USE_MALLOC=y
```

或通过 `idf.py menuconfig` → `Component config` → `ESP PSRAM` 启用。

启用后 `malloc()` 自动从 8MB PSRAM 分配，GIF 28 帧和 Marquee 精灵均能正常创建。

## 3. pushSprite 透明色注意事项

```cpp
sprite.pushSprite(&tft, x, y, COLOR_BLACK);
```

- 第 4 个参数是**透明色**，该颜色的像素不会被绘制
- 对 RGB565 sprite，直接传 RGB565 值（如 `0x0000` = 黑色）
- 若 sprite 内容含该颜色，这些区域不显示（在黑色背景上通常无视觉差异）
- 若不想透明：省略第 4 个参数或传 `~0u`

## 4. 边缘检测按键（替代时间门控防抖）

### 问题

传统 `millis() - last > 200ms` 防抖在系统卡顿（绘图 100~200ms）时，弹窗确认需要按两次。

### 解决方案：边沿检测状态机

```cpp
static int prev_btn = BTN_NONE;

static int read_buttons() {
    int curr = BTN_NONE;
    if (gpio_get_level(BTN_UP) == 0)    curr = BTN_U;
    if (gpio_get_level(BTN_DOWN) == 0)  curr = BTN_D;
    // ...

    int event = (curr != BTN_NONE && curr != prev_btn) ? curr : BTN_NONE;
    prev_btn = curr;
    return event;
}
```

特性：长按仅触发一次；松手再按立即响应；不受系统延迟影响。

### 盲区补偿

长时间绘图后需调用 `sync_button_state()` 重新校准：

```cpp
static void sync_button_state() {
    prev_btn = BTN_NONE;
    if (gpio_get_level(BTN_UP) == 0)       prev_btn = BTN_U;
    else if (gpio_get_level(BTN_DOWN) == 0) prev_btn = BTN_D;
    // ...
}
```

在每个 `draw_*` 函数末尾调用。

## 5. ESP-IDF vs Arduino 关键差异

| 方面               | Arduino                           | ESP-IDF (本项目)                   |
| ------------------ | --------------------------------- | ---------------------------------- |
| SPIFFS 访问        | `SPIFFS.open()` 对象            | VFS`/spiffs/` 路径 + `fopen()` |
| LovyanGFX 图片加载 | `drawPngFile(SPIFFS, "/x.png")` | 不支持，需 FILE-buffer 方案        |
| 时间函数           | `millis()`                      | `esp_timer_get_time() / 1000`    |
| GPIO 读取          | `digitalRead()`                 | `gpio_get_level()`               |
| PSRAM 配置         | `board_build.psram`             | `CONFIG_SPIRAM=y` in sdkconfig   |

## 7. IMG 浏览器每帧重载 Bug（已修复）

### 现象

进入 IMG 浏览器后，串口反复打印 `IMG loading: /spiffs/0001.png`，图片不断重载，PSRAM 快速耗尽。

### 根因

`handle_img()` 的 `switch` 语句中 `default: break` 后无条件调用 `draw_img_browser()`，主循环每 10ms 触发一次无按键重绘，每次都重新 `fopen`、`malloc`、解码整个 PNG。

### 修复

```cpp
default:
    if (btn == BTN_NONE) return;  // 无按键不重绘
    break;
```

## 8. 资源路径迁移到 SD 卡

所有图片、GIF 帧、走马灯数据、音乐文件已从 SPIFFS 迁移到 SD 卡：

| 资源 | 旧路径 | 新路径 |
|------|--------|--------|
| 图片 | `/spiffs/0001.png` | `/sdcard/0001.png` |
| GIF 帧 | `/spiffs/gif_0001.png` | `/sdcard/gif_0001.png` |
| 走马灯 | `/spiffs/500x150.raw` | `/sdcard/500x150.raw` |
| 音乐 | `/spiffs/music.wav` | `/sdcard/music.wav` |

SD 卡文件需提前通过 PC 拷贝到位（目录结构见 [SD卡开发指南](SD卡开发指南.md)）。
